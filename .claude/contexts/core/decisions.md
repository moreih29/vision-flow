# Design Decisions

기존 레포에서 기록된 설계 결정 사항(D001~D024)을 재검토하고 정리한 문서.
각 결정의 맥락, 이유, 트레이드오프를 기록한다.

## 스토리지 & 인프라

### D001: 로컬 디스크 스토리지 (MinIO 미사용)
- **결정**: 기본 스토리지로 로컬 파일시스템 사용
- **이유**: MinIO 컨테이너가 GPU 학습과 CPU/메모리를 경쟁. 개발 단순성 우선.
- **트레이드오프**: 단일 서버에 한정. 향후 S3 인터페이스로 교체 가능하도록 추상화 레이어 구축.
- **유지**: O (StorageBackend ABC로 확장성 확보 완료)

### D002: Docker Compose는 인프라만
- **결정**: PostgreSQL + Redis만 Docker로 실행, 애플리케이션은 호스트에서 직접
- **이유**: 개발 중 HMR/핫 리로드를 위해 호스트 실행이 편리. 컨테이너 빌드 오버헤드 제거.
- **유지**: O (프로덕션에서는 별도 Dockerfile 작성 예정)

## 프론트엔드

### D003: 자체 라벨링 도구 (Konva.js)
- **결정**: Label Studio 등 외부 도구 대신 Konva.js로 자체 구현
- **이유**: 데이터 파이프라인 완전 통합, AI 보조 라벨링 구현 용이, 커스터마이징 자유도
- **트레이드오프**: 개발 공수 증가. 그러나 장기적으로 UX 최적화와 기능 확장에 유리.
- **유지**: O

### D017: 배치 업로드 20개 제한
- **결정**: 파일 업로드 시 20개씩 분할하여 순차 전송
- **이유**: 서버/브라우저 메모리 보호. 대량 파일 동시 전송 시 OOM 방지.
- **유지**: O (숫자는 실험으로 조정 가능)

### D018: 이미지 파일 서빙 FileResponse
- **결정**: 정적 파일 서빙이 아닌 API 엔드포인트(FileResponse)로 이미지 제공
- **이유**: JWT 인증 적용 필요. 이미지별 접근 권한 제어. `<img src>`에서 쿼리 파라미터 토큰 사용.
- **트레이드오프**: CDN 캐싱 어려움. 대량 이미지 로딩 시 성능 병목 가능.
- **유지**: O (보안 우선. 성능은 향후 CDN/프록시로 보완)

### D019: 가상화 스크롤 (@tanstack/react-virtual)
- **결정**: 이미지 목록에 가상화 스크롤 적용
- **이유**: 수천~수만 이미지 렌더링 시 DOM 성능 저하 방지
- **유지**: O (필수 최적화)

### D022: 브라우저 기본 UI 금지
- **결정**: `window.confirm()`, `window.alert()` 대신 shadcn AlertDialog 사용
- **이유**: 일관된 UX, 스타일링 가능, 비동기 처리 용이
- **유지**: O

### D023: 로그인 정보 저장
- **결정**: JWT 토큰 7일 유효, localStorage에 저장
- **이유**: 매번 로그인하는 불편함 해소
- **보안 검토**: XSS 취약점 존재 시 토큰 탈취 가능. httpOnly 쿠키 전환 고려.
- **유지**: △ (보안 강화 시 쿠키 기반으로 전환 검토 필요)

## 데이터 관리

### D007: 하이브리드 데이터 버전 관리
- **결정**: 내부적으로는 해시 중복 제거, 사용자에게는 스냅샷 버전 제공
- **이유**: 디스크 효율성(내부)과 직관적 UX(외부)의 양립
- **유지**: O

### D012: DataStore + Task 이원 구조
- **결정**: 원본(DataStore)과 라벨링/학습 작업(Task)을 분리
- **이유**: 원본 데이터의 자유로운 관리와 학습용 데이터의 체계적 관리를 양립
- **변경 이력**: 초기 "Data Pool + Subset" 구조에서 D025에 의해 리네이밍
- **유지**: O (핵심 설계)

### D013: Task 생성 시 Task Type 필수, 변경 불가
- **결정**: Task의 task_type은 생성 시 확정, 이후 변경 불가
- **이유**: Task Type이 라벨 포맷을 결정. 변경 시 기존 라벨 호환성 깨짐.
- **유지**: O

### D014: 수동 스냅샷 버전 관리
- **결정**: 사용자가 명시적으로 버전 생성 (자동 아님)
- **이유**: 의미 있는 시점에만 버전 기록. 자동 버전은 너무 빈번할 수 있음.
- **유지**: O

### D015: 클래스 스키마 해시
- **결정**: 클래스 구성의 SHA-256 해시로 호환성 검사
- **이유**: 서로 다른 Subset 간, 또는 버전 간 클래스 호환성 자동 판단
- **유지**: O

### D016: 시맨틱 모델 버저닝
- **결정**: vX.Y.Z (X=클래스, Y=데이터, Z=설정)
- **이유**: 모델 변경의 의미를 버전 번호에 인코딩. 호환성 판단 용이.
- **유지**: O

## 백엔드

### D004: SSE(실시간) + REST(히스토리) 이중 채널
- **결정**: 학습 모니터링에 SSE, 학습 기록에 REST API
- **이유**: 실시간 스트리밍과 과거 기록 조회는 다른 패턴. 적합한 프로토콜로 분리.
- **유지**: O

### D020: Alembic 비동기 마이그레이션
- **결정**: Alembic을 async 모드로 실행
- **이유**: asyncpg 드라이버 사용에 맞춰 마이그레이션도 비동기로 통일
- **유지**: O

### D021: 서비스 싱글톤 패턴
- **결정**: 서비스를 모듈 레벨 싱글톤으로 생성 (`service = Service()`)
- **이유**: DI 컨테이너 없이 간단하게 의존성 관리. 현재 규모에서 적합.
- **검토**: 프로젝트 규모 확대 시 DI 프레임워크 도입 고려
- **유지**: △ (규모에 따라 재검토)

## 보안

### D024: CORS 설정
- **결정**: 환경변수 `CORS_ORIGINS`로 허용 origin 제어 (기본값 `http://localhost:5273`)
- **이유**: 개발 편의성과 보안 양립. 프로덕션에서는 환경변수만 변경하면 됨.
- **변경 이력**: 초기 `allow_origins=["*"]` → 환경변수 기반으로 전환
- **유지**: O

## 도메인 구조

### D025: Dataset→DataStore, Subset→Task 리네이밍
- **결정**: `Dataset`을 `DataStore`로, `Subset`을 `Task`로 이름 변경. Task에 `status` 컬럼 추가.
- **이유**:
  - "Subset"이라는 이름이 실제 역할(독립적 작업 단위)과 불일치
  - "Task"가 ML 워크플로에서 보편적 용어 (classification task, detection task)
  - "Database"는 RDBMS와 혼동되므로 "DataStore" 채택
  - Phase 4(Training), Phase 5(Pipeline)에서 Task가 자연스럽게 학습/추론의 단위가 됨
- **영향 범위**: Backend 모델/스키마/서비스/라우터 전체, Frontend 타입/API/페이지/컴포넌트 전체, Alembic 마이그레이션
- **유지**: O

### D026: React Query 도입
- **결정**: `@tanstack/react-query`로 서버 상태 관리, `useState+useEffect` 패턴 대체
- **이유**: 캐싱, 자동 무효화, 로딩/에러 상태 중복 제거. 프로젝트/태스크/DataStore API에 적용.
- **트레이드오프**: 새 의존성 추가. 그러나 코드 중복 대폭 감소 및 UX 개선 (staleTime 캐싱).
- **유지**: O

### D027: 공통 에러 핸들러 + 구조화된 로깅
- **결정**: FastAPI exception_handler로 모든 예외를 표준 JSON 형식으로 변환. Python 표준 logging으로 JSON 로그 출력.
- **이유**: 에러 응답 형식 일관성, 스택트레이스 사용자 노출 방지, 로그 파싱/분석 용이.
- **유지**: O

### D028: 전 엔드포인트 소유권 검증 (IDOR 방지)
- **결정**: 모든 API 엔드포인트에서 `check_ownership` 호출로 리소스 소유자 검증
- **이유**: 인증만으로는 불충분. 다른 사용자의 리소스 접근을 원천 차단.
- **유지**: O (보안 필수)

### D029: 더블 서밋 방지 (useRef 가드)
- **결정**: 생성/삭제 핸들러에 `creatingRef`/`deletingRef`로 동기적 가드 적용
- **이유**: React의 `setState`는 비동기(배치)라 `disabled` 상태가 즉시 반영되지 않음. `useRef`로 동기적 중복 호출 차단.
- **유지**: O (모든 mutate 핸들러에 적용 원칙)

## 결정 기록 원칙

1. 결정의 배경/맥락을 명확히 기록
2. 대안과 트레이드오프를 함께 기록
3. "유지" 상태를 정기적으로 검토 (O: 유지, △: 재검토 필요, X: 변경됨)
4. 새로운 결정은 D025부터 순차 번호 부여
